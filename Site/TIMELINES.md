# VISCERIUM timeline system

The Codex supports two related Chronos workflows:

1. **Canonical generated timelines** compile structured event and era notes into `super`, `citadel`, `smog`, `nearsight` and `entropy` datasets.
2. **Native Chronos timelines** use ordinary fenced `chronos` Markdown blocks for fast note-local comparisons, research, scene chronology and editorial sketches.

Both render through `chronos-timeline-md` in Astro/Starlight. The installed public Chronos Obsidian plugin renders native blocks locally, while the maintained VISCERIUM Obsidian plugin renders canonical `[Timeline:...]` shortcodes from the shared calendar/compiler model.

## Architecture

```text
Canonical path
──────────────
Vault canon notes
  calendarDate / calendarEndDate
          │
          ▼
Site/src/lib/calendar/runtime.mjs
  registered calendars ↔ absolute world-day
          │
          ▼
Site/src/lib/timeline/compiler.mjs
  validation, era membership, sorting, dataset generation
          │
          ▼
Site/src/lib/timeline/chronos-adapter.mjs
  normalized events → Chronos ParseResult
          │
          ▼
Site/src/lib/timeline/chronos-renderer.mjs
  Chronos core + VISCERIUM controls, calendar axis and minimap

Native Markdown path
────────────────────
chronos fenced block in a Vault note
          │
          ├── public Chronos plugin → Obsidian Reading view
          └── sync transform → ChronosEmbed.astro → Starlight
```

`Site/src/lib/timeline/renderer.mjs` is a compatibility entry point that re-exports the Chronos renderer.

Chronos remains backed by `vis-timeline`; VISCERIUM no longer constructs the principal timeline directly. It supplies Chronos with structured parsed data, then uses the public underlying timeline instance for host features such as filters, custom calendars, minimap synchronisation and URL state.

## Canonical chronology rule

`calendarDate` is the only canonical event start date. Never add `timeline.year`, `timeline.date` or `timeline.id`. The compiler rejects those legacy fields.

The compiler converts dates to absolute world-days. The Chronos adapter maps those days to synthetic UTC JavaScript dates solely for positioning. One world day is exactly one synthetic UTC day. Native Gregorian labels are hidden on canonical timelines and replaced with labels generated by the selected VISCERIUM calendar.

A native `chronos` block is a self-contained presentation timeline. Its date text is its own source and is not automatically imported into the canonical compiler. Do not duplicate a canonical event in a Chronos block merely to place it on a generated era timeline.

## Canonical event schema

```yaml
---
title: Example Event
description: "A concise public description."
publish: true
status: canon
slug: eras/citadel/events/example-event
type: event
era: CITADEL

calendarDate:
  calendar: okse
  year: 9250
  month: niewmonath
  day: 1
  precision: day
  certainty: exact

calendarEndDate:

timeline:
  kind: event
  importance: standard
  categories:
    - military
    - political
  lanes:
    - okse-dominion
  global: auto
  era: auto
  order: 10

location:
  - Example Location
faction:
  - Okse Dominion
participants:
  - Example Character
tags:
  - example
---
```

Optional fields may be omitted. A note with `calendarEndDate` defaults to `period`; a point defaults to `event`. `timeline.order` is only a same-day tie-breaker.

### Date presentation

`precision` controls labels, never sorting or position:

- `day`
- `month`
- `year`

`certainty` controls visual presentation, never position:

- `exact`
- `approximate`
- `disputed`
- `legendary`

Start and end dates may use different registered calendars. Both resolve independently to absolute world-days.

### Importance

- `landmark`
- `major`
- `standard`
- `minor`
- `incidental`

`landmark` and `major` enter the `super` timeline automatically. Override with `timeline.global: include` or `exclude`.

### Categories and lanes

Categories are open-ended. Known visual mappings exist for technology, military, political, cultural, religious, economic, scientific, disaster, resonance, myrkild, naranor, exploration, social and environmental. Unknown categories use a neutral fallback.

Lanes are optional identifiers for factions, nations, regions, organisations or story threads. Readers can switch between unified, declared-lane and category views. The adapter caps the visible group count and folds low-volume groups into **Other / unassigned** before creating Chronos groups.

## Canonical era schema

```yaml
---
title: CITADEL
description: "A time of steel, bone and thrones."
publish: true
status: canon
slug: eras/citadel
type: era
eraId: citadel

calendarDate:
  calendar: okse
  year: 9201
  month: niewmonath
  day: 1
  precision: year
  certainty: exact

calendarEndDate:
  calendar: okse
  year: 9400
  intercalaryDay: engimanutur-02
  precision: year
  certainty: exact

timeline:
  kind: era
  order: 1
  visualToken: e1
  allowGapAfter: true
  defaultViewport:
    paddingDays: 56
---
```

Canonical era records live in `Vault/Lore/Eras/`. Point events belong to the era containing their absolute day. Periods belong to every era they overlap. A declared `era` that disagrees with chronology fails validation. Set `allowGapAfter: true` only when a historical gap is deliberate.

## Generated timelines and shortcodes

The compiler always generates `super`, `citadel`, `smog`, `nearsight`, `entropy` and `manifest.json` under `Site/src/data/timelines/`. These files are build outputs.

Simple form:

```md
[Timeline:super]
```

Inline form:

```md
[Timeline:super calendar=okse lane=unified filters=true minimap=true]
```

Configured form:

```yaml
timelineBlocks:
  ID-0001:
    timeline: super
    defaultCalendar: okse
    laneMode: unified
    showFilters: true
    showMinimap: true
    showLegend: true
```

```md
[Timeline:ID-0001]
```

The sync step converts the shortcode to `TimelineEmbed.astro`. The VISCERIUM Obsidian plugin detects the same shortcode in Reading view and compiles local vault records without requiring the website server.

## Native Chronos Markdown

For a quick timeline directly in a note, use the syntax supported by the public Chronos plugin:

````md
```chronos
> NOTODAY
> ORDERBY start
> HEIGHT 360

- [9201] An event | Hover detail
- [9204~9208] A ranged event
@ [9201~9400] #gray CITADEL
* [9210] A point
= [9220] A marker
- [9230] #red {Okse Dominion} Grouped event
```
````

The public Chronos plugin renders this block in Obsidian. During site sync, `transform-timeline-shortcodes.mjs` converts the same fence to `ChronosEmbed.astro`, preserving Chronos events, periods, points, markers, groups, colours, flags, drag/zoom interaction, tooltips and the fit-all control in Starlight.

Native Chronos dates use Chronos's numeric ISO-like syntax. They do not use the registered fictional-calendar parser. Use native blocks for local or editorial timelines; use canonical event notes when the event must participate in calendar switching, era validation, filters or generated timelines.

`Vault/Templates/Chronos Timeline Template.md` provides a ready-to-edit example.

## Adding a canonical event

1. Create the note beneath `Vault/Lore/` using `Vault/Templates/Event Template.md`.
2. Set `publish: true`, `status: canon`, `type: event` and a concise description.
3. Author exactly one `calendarDate`.
4. Add `calendarEndDate` only for a range.
5. Choose importance, categories and optional lanes.
6. Declare the expected top-level era. The compiler verifies it from chronology.
7. Run `npm run sync` and `npm run generate:timelines` from `Site/`.

## Obsidian setup

Install and enable both plugins for the complete authoring experience:

- **Chronos Timeline** from the Obsidian community plugins directory: renders fenced `chronos` blocks and provides its templates/commands.
- **VISCERIUM Timelines** from `Tools/obsidian-viscerium-timelines`: renders `[Timeline:...]` shortcodes from canonical vault metadata.

The two plugins have separate responsibilities and can coexist. The VISCERIUM plugin does not register the `chronos` code-block language, avoiding duplicate processors.

## Site commands

From `Site/`:

```bash

npm run test:unit

```

`chronos-timeline-md` is pinned to `1.1.0`. `vis-timeline` remains pinned because it is Chronos's peer rendering engine and is also used by the lightweight overview timeline.

## Performance model

- Only the selected generated dataset is bundled into a canonical timeline page.
- Article bodies are not copied into generated timeline JSON.
- At distant zoom levels, lower-importance events are omitted before the Chronos item set is updated.
- Calendar labels are reformatted from absolute days without changing Chronos positions.
- Lane/category grouping is capped.
- Canonical pages contain a static chronological fallback list.
- Native Chronos blocks ship only their own Markdown source.

Run `npm run benchmark:timelines` to compile deterministic synthetic sets of 1,000 and 5,000 events. The compiler benchmark remains independent of the browser renderer.

## Troubleshooting

- **Legacy field error:** remove `timeline.id`, `timeline.year` or `timeline.date`; move the date into `calendarDate`.
- **Unknown month/intercalary day:** use the slug from the source calendar definition.
- **Era mismatch:** correct the event date or declared top-level era. Do not override calculated membership.
- **Period missing end:** add `calendarEndDate` or change `timeline.kind`.
- **Native Chronos block renders in Obsidian but not the site:** run `npm run sync`; confirm the fence language is exactly `chronos`.
- **Canonical shortcode does not render in Obsidian:** build and enable the VISCERIUM Timelines plugin; the public Chronos plugin only owns fenced blocks.
- **Blank interactive view:** the canonical static list or native source fallback should remain visible. Check the browser console and reinstall dependencies from the current lockfile.
- **Obsidian article link does not open:** ensure the compiled event's source note still exists beneath `Lore/`.
